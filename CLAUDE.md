## セキュリティ（最重要）

### 絶対禁止事項
以下のルールは、情報漏洩を防ぐために最も重要なルールです。

- **APIキー・トークン・パスワードは絶対にコードにハードコードしない**
  - APIキー = 外部サービスに接続するためのパスワードのようなもの
  - コードに直接書くと、GitHubを通じて外部に漏洩する危険がある
- **settings.jsonのallow listにトークンを含むコマンドを個別登録しない**
  - ワイルドカード `Bash(curl:*)` 等を使い、トークンは環境変数から読む
- **同じAPIキーを複数プロジェクトで使い回さない**
  - 1つ漏洩すると全プロジェクトが危険にさらされるため
- **GASコード内にシークレットを直書きしない**（`const TOKEN = '...'` は禁止）
- **Webhook URLもシークレットとして扱う**（URLにキー情報が含まれるため）

### シークレット（秘密情報）の保存場所
シークレット = APIキー、トークン、パスワードなどの秘密情報のこと。

| 環境 | 保存先 | 備考 |
|-----|-------|------|
| ローカル開発 | `.env` ファイル | `.gitignore` に含まれていることを必ず確認 |
| Cloud Run | GCP Secret Manager にマウント | |
| GAS | `PropertiesService.getScriptProperties()` | GASエディタ > プロジェクトの設定 で手動設定 |

### APIキーの発行・管理ルール
- **粒度**: プロジェクトごとに個別に発行する（使い回し禁止）
- **命名規則（GCP Console）**: `{プロジェクト名}-{サービス名}`（例: `seo-writing-gemini`）
- **API制限**: 発行時に必要なAPIのみに制限する（例: Gemini API のみ）
- **スクリプトプロパティ名**: `GEMINI_API_KEY`, `CHATWORK_API_TOKEN` 等、サービス名ベースで統一
- **ローテーション**: 漏洩が疑われたら即時変更。定期的には半年に1回

### GCPプロジェクトの使い分け
| 用途 | GCPプロジェクト | プロジェクトID |

| 会社の業務用 | Collabohouse Automation | `collabohouse-automation` |

### コミット時の安全確認
- **git commitする前に差分を確認し、シークレットが含まれていないことを確認する**
- secret-guard.sh hookが自動チェックするが、過信せず目視も行う
- **検出パターン**: `AIzaSy`, `AKIA`, `sk-`, `ghp_`, `xoxb-`, `GOCSPX-`, `1//`, `ya29.`, `sk-ant-`

## GCP全般
- **GCPプロジェクトの紐づけは勝手に決めない**。必ず「どのGCPプロジェクトに紐づけますか？」と確認すること
- GCPへのデプロイ時、DockerイメージのビルドはDocker Desktopを使わず、必ず**Cloud Build**を使うこと
- GCPでジョブやサービスを実装したら、必ず**Cloud Monitoringでエラー通知を設定**すること
  - Cloud Run Jobの場合: Completed Executions / result=failed / count > 0 → メール通知
- Cloud Run Job実行時は、初回実行後にログを確認して**メモリ不足（OOM）やクラッシュが起きていないかチェック**すること

## 開発ルール

### Excel出力
- フォントは **Meiryo UI** 固定

### 自動化・開発
- 自動化処理の実行場所（ローカル vs Cloud Run等）は勝手に決めない。必ず**確認してから決める**こと
- 外部サービス連携（API・Drive・GCS等）は、本番に組み込む前に必ず**小さなテストで接続確認**する
- GAS・GCPの操作はclasp / Apps Script API等を使って**Claude自身が実行する**こと。ユーザーに手動操作させない
  - ただし、GCPプロジェクト・スコープ・API有効化などの設定内容は確認しながら進めること
- ブラウザ自動操作は**Playwright**を使うこと（Puppeteerは使わない）

## Hooks（自動ガードレール）
Hooksとは、Claude Codeが危険な操作をしようとしたときに自動でブロックする仕組みです。
すべてのプロジェクトに設置されています。

### 実行前チェック（PreToolUse）
- **block-dangerous.sh** — `rm -rf`（全削除）, `git push --force`（強制上書き）等の破壊コマンドをブロック
- **secret-guard.sh** — `git commit` 時にステージされた差分にAPIキーやトークンが含まれていないかチェック

### 実行後チェック（PostToolUse）
- **verify-gas-deploy.sh** — `clasp push`（GASデプロイ）後に自動で結果を検証。認証切れ・エラーを検出し、対処法を通知
- **verify-xlsx.sh** — Excelファイル出力後にファイルの整合性を自動検証。破損していればブロック

新プロジェクト作成時にこれらのhookは自動でコピーされます。

## settings.jsonのallow list方針
- コマンド系は**ワイルドカード**を使う: `Bash(curl:*)`, `Bash(node:*)` 等
- 個別のURLやトークン付きコマンドを登録しない
- MCP系もワイルドカードで: `mcp__google-calendar__*` 等
